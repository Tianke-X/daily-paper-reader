## 摘要
现有的嵌入模型本质上是静态的：它们孤立地对文本片段进行编码，忽略其周围的上下文和时间顺序。本文提出了 EvoEmbedding，一种新颖的嵌入模型，能够为检索生成可演化的表示。它专为长上下文场景定制，在此类场景中，信息是动态的、顺序的，并且需要连续的状态跟踪。我们的设计很简单：EvoEmbedding 在顺序处理输入时维护一个持续更新的潜在记忆，并将其与原始内容一起用于联合生成可演化的嵌入。因此，对于相同的查询，我们的模型会根据不断变化的上下文调整其表示以检索不同的目标，超越了静态的语义搜索。为使模型具备此能力，我们构建了 EvoTrain-180K，一个用于潜在记忆和检索联合优化的多样化数据集。此外，我们引入了一个记忆队列来防止循环编码期间的表示崩溃，同时采用分片段批处理技术来解决显著的长度差异，并将训练加速了 3.8 倍。大量实验表明，我们的模型不仅在一系列长上下文检索基准上超越了更大规模的专业模型（例如 Qwen3-Embedding-8B 和 KaLM-Embedding-Gemma3-12B），还能很好地泛化到上下文比其训练窗口长 10 倍的下游任务（例如个性化）。值得注意的是，EvoEmbedding 能够无缝集成到智能体工作流程中以提升性能。例如，一个配备了我们的模型的朴素 RAG 流水线超越了专门的智能体记忆系统。项目页面：https://clare-nie.github.io/EvoEmbedding。

## Abstract
Existing embedding models are inherently static: they encode text segments in isolation, ignoring their surrounding context and temporal order. This paper introduces EvoEmbedding, a novel embedding model that generates evolvable representations for retrieval. It is tailored for long-context scenarios, where information is dynamic, sequential, and requires continuous state tracking. Our design is simple: EvoEmbedding maintains a continuously updated latent memory as it sequentially processes inputs, and uses it alongside the raw content to jointly generate evolvable embeddings. Consequently, for the same query, our model adapts its representation to retrieve distinct targets based on the evolving context, going beyond static semantic search. To equip the model with this capability, we construct EvoTrain-180K, a diverse dataset for the joint optimization of latent memory and retrieval. Furthermore, we introduce a memory queue to prevent representation collapse during recurrent encoding, alongside segment-batching techniques that tackle significant length variance and accelerate training by 3.8$\times$. Extensive experiments show that our model not only outperforms larger-scale specialists (e.g., Qwen3-Embedding-8B and KaLM-Embedding-Gemma3-12B) across a range of long-context retrieval benchmarks, but also generalizes well to downstream tasks (e.g., personalization) with contexts 10$\times$ longer than its training window. Notably, EvoEmbedding seamlessly integrates into agentic workflows to boost performance. For instance, a naive RAG pipeline equipped with our model surpasses dedicated agentic memory systems. - **整体含义**：EvoEmbedding 力图在**表示层面**解决缺陷——通过引入持续更新的潜在记忆，使同一查询在不同上下文下能检索到不同目标，实现真正的上下文感知检索，从而统一提升长上下文检索和智能体记忆系统的效果。

## 2. 论文提出的方法论

### 核心思想
- **潜在记忆（Latent Memory）**：维护一个固定容量的 FIFO 队列作为潜在记忆，顺序处理输入片段时不断更新。嵌入生成不仅依赖当前输入，还**联合利用历史记忆**，从而注入时序动态和跨片段关联。
- **并行任务**：每个步骤同时执行记忆演化（更新记忆）和表示生成（输出可演化嵌入）。

### 关键技术细节
1. **记忆演化**  
   - 激活记忆 LoRA 适配器，将历史记忆投影为 `[memory; input; learnable tokens]` 输入 LLM，生成新的 K 个潜在 token，投影后更新 FIFO 记忆队列（容量 C=512 tokens）。
2. **表示生成**  
   - 激活检索 LoRA 适配器，将历史记忆投影、当前输入和 `<EOS>` 组合输入 LLM，取最后一个 token 的隐状态投影为最终 embedding（维度 Demb=1024）。
3. **训练损失**  
   - \( \mathcal{L} = \mathcal{L}_{\text{mem}} + \mathcal{L}_{\text{con}} \)  
   - \( \mathcal{L}_{\text{mem}} \)：使用记忆 + 查询预测答案的交叉熵（冻结 backbone，仅优化记忆模块）。  
   - \( \mathcal{L}_{\text{con}} \)：长度加权的多正例对比损失，对数项含 `log(N+1)` 以自适应不同上下文长度。
4. **动态分段批处理（Segment-Batching）**  
   - 将 k 个连续片段拼接到一条样本（总长度 ≤ 2048 tokens）并行处理，避免逐片段串行，训练加速 3.8 倍。
5. **EvoTrain-180K 数据集**  
   - 三阶段自动流水线：① 构建多种上下文（网页、对话、记忆）；② 用 40+ 模板和多种 LLM 生成含简单/复杂问题的 QA 对；③ 用 Gemini-3.1-Pro 标注正负样本并过滤。共 184,137 条样本，最大 12K tokens，256 片段。

### 算法流程（文字描述）
- **记忆演化**：输入当前片段 xt 和记忆 Mt−1 → 激活记忆 LoRA → 拼接记忆投影、xt 和可学习 token → LLM 前向 → 输出新潜在 token → 投影后入 FIFO 队列 → 更新记忆。
- **嵌入生成**：输入待编码文本（片段或查询）和记忆 Mt−1 → 激活检索 LoRA → 拼接记忆投影、文本和 `<EOS>` → LLM 前向 → 取最后隐状态 → 投影为 embedding。

## 7. 优点

- **设计简洁高效**：仅用潜在记忆队列 + 联合编码，就从根本上解决了静态嵌入的局限性，无需复杂流水线。
- **训练成本低**：仅需 180K 合成数据（不到典型模型 1% 的数据量），标准 SFT 即可达到 SOTA，数据生成流水线可自动化。
- **参数效率高**：多 LoRA 设计避免灾难性遗忘，低秩（r=16）即可保持稳定性能，方便部署。
- **零额外 token 开销**：潜在记忆在编码时构建，推理时无需额外存储或生成，优于所有需要显式记忆构造的系统。
- **兼容性与可扩展性**：可作为独立嵌入模型或即插即用模块，无缝融入现有 RAG 和智能体系统。
- **完备的消融与敏感性分析**：验证每个设计选择（记忆队列、分段批处理、长度加权、容量、秩）的必要性和最优配置。

## 8. 不足与局限

- **计算资源与数据规模**：受限于单台 8×H800 和 180K 数据，模型在高度域外场景（如极长文档、罕见领域）性能可能下降。
- **不支持多模态**：当前仅处理文本，无法处理图像、音频等多模态长上下文。作者承认扩展到多模态需要更大记忆队列和维度对齐。
- **编码延迟**：由于顺序处理上下文，编码时间显著长于静态并行模型（22.08s vs 5.52s），不适合对延迟极敏感的场景（如实时对话）。但峰值内存更低 (20.9GB vs 43.1GB) 可作为权衡。
- **泛化验证有限**：虽然测试了 128K 上下文，但训练窗口仅 12K，泛化到更长的上下文（如 1M）尚未验证。
- **潜在记忆容量固定**：C=512 可能不足以捕获极长对话中的所有关键信息，尽管实验表明性能在 512 处饱和。
- **缺少多语言/跨语言评估**：数据集主要为英文，未评估中文或其他语言的性能。
- **生成任务评估依赖自动评判**：使用 GPT-4o-mini 而非人工，可能存在偏差；LoCoMo 上的 Temporal 子任务仍落后于 LightMem（49.8% vs 76.3%），说明时间序列推理仍有改进空间。
