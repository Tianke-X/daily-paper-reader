## 摘要
现有的嵌入模型本质上是静态的：它们孤立地编码文本片段，忽略其周围上下文和时间顺序。本文介绍了EvoEmbedding，一种新颖的生成可演化表示用于检索的嵌入模型。它专为长上下文场景设计，其中信息是动态、顺序的，并需要连续状态跟踪。我们的设计简单：EvoEmbedding在顺序处理输入时维护一个持续更新的潜在记忆，并利用它与原始内容一起共同生成可演化嵌入。因此，对于相同的查询，我们的模型会根据演化的上下文调整其表示以检索不同的目标，超越静态语义搜索。为了使模型具备这种能力，我们构建了EvoTrain-180K，一个用于潜在记忆和检索联合优化的多样化数据集。此外，我们引入了记忆队列以防止循环编码过程中的表示崩溃，以及处理显著长度变化并将训练加速3.8倍的分段批处理技术。大量实验表明，我们的模型不仅在多个长上下文检索基准中优于更大规模的专业模型（如Qwen3-Embedding-8B和KaLM-Embedding-Gemma3-12B），而且在上下文长度为其训练窗口10倍的下游任务（如个性化）中也能良好泛化。值得注意的是，EvoEmbedding无缝集成到代理工作流中以提升性能。例如，配备我们模型的朴素RAG流水线超越了专门的代理记忆系统。项目页面：https://clare-nie.github.io/EvoEmbedding/。

## Abstract
Existing embedding models are inherently static: they encode text segments in isolation, ignoring their surrounding context and temporal order. This paper introduces EvoEmbedding, a novel embedding model that generates evolvable representations for retrieval. It is tailored for long-context scenarios, where information is dynamic, sequential, and requires continuous state tracking. Our design is simple: EvoEmbedding maintains a continuously updated latent memory as it sequentially processes inputs, and uses it alongside the raw content to jointly generate evolvable embeddings. Consequently, for the same query, our model adapts its representation to retrieve distinct targets based on the evolving context, going beyond static semantic search. To equip the model with this capability, we construct EvoTrain-180K, a diverse dataset for the joint optimization of latent memory and retrieval. Furthermore, we introduce a memory queue to prevent representation collapse during recurrent encoding, alongside segment-batching techniques that tackle significant length variance and accelerate training by 3.8$\times$. Extensive experiments show that our model not only outperforms larger-scale specialists (e.g., Qwen3-Embedding-8B and KaLM-Embedding-Gemma3-12B) across a range of long-context retrieval benchmarks, but also generalizes well to downstream tasks (e.g., personalization) with contexts 10$\times$ longer than its training window. Notably, EvoEmbedding seamlessly integrates into agentic workflows to boost performance. For instance, a naive RAG pipeline equipped with our model surpasses dedicated agentic memory systems. ## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：现有嵌入模型是静态的：它们孤立地编码文本片段，忽略上下文和时间顺序。在长上下文、动态、顺序信息场景（如对话历史、多轮交互）中，静态语义搜索无法适应状态变化，例如用户更改会议时间后依然检索到旧信息。
- **整体含义**：静态嵌入+分段检索的范式破坏了时间连续性和跨段关联，导致检索不准确。传统的RAG流水线虽然通过索引优化（如知识图谱、元数据）或查询重写/重排序来缓解，但引入了额外开销，仍不理想。论文旨在从**表示层面**解决这一问题，让嵌入随上下文演化。

## 2. 方法论
- **核心思想**：EvoEmbedding 在顺序处理输入时维护一个**持续更新的潜在记忆**，将其与原始内容联合生成**可演化的嵌入**。对于同一查询，在不同上下文中能检索到不同目标。
- **关键技术细节**：
  - **双分支并行**：给定当前输入段 `x_t` 与前一记忆 `M_{t-1}`，同时执行：
    - 记忆演化（Memory Evolution）：生成新记忆 `~M_t`，通过 FIFO 队列更新为 `M_t`。
    - 表示生成（Representation Generation）：结合当前段和记忆生成检索向量 `v_t`。
  - **潜在记忆队列**：使用容量为 `C = 512` 的 FIFO 队列（每个步长生成 `K=16` 个潜在 token）。好处：① 有限循环编码，避免表示崩溃；② 固定容量，限制计算开销（≈编码一张图片）。
  - **动态分段批处理（Segment-Batching）**：将 `k` 个连续段拼接到一起并行处理（总长度不超过阈值如2048 token），加速训练 3.8×，且提升性能。
  - **多 LoRA 设计**：记忆演化和表示生成使用不同的 LoRA 适配器，冻结基座 LLM，避免灾难性遗忘，可灵活切换。
- **训练损失**：联合优化两个损失：
  - 记忆生成损失 `L_mem`：用交叉熵预测目标答案（冻结基座，隐含地迫使记忆兼容基座语义空间）。
  - 对比表示损失 `L_con`：长度加权的多正例对比损失，动态平衡不同长度样本难度。
- **数据集 EvoTrain-180K**：自动化三阶段合成管道构建，含 184k 样本，最大长度 12K tokens，涵盖多领域（Web文本、对话、记忆），包含简单语义匹配和复杂推理 QA。

## 6. 主要结论与发现
- EvoEmbedding 在**检索**上达到 SOTA：比 KaLM-12B 高 6.4%（Recall@10），比 Qwen3-Embedding-8B 高 11.1%。
- 在**生成任务**上，朴素 RAG + EvoEmbedding 超越了专门的记忆系统（如 LongMemEval 上 77.6% vs LightMem 70.2%）。
- 模型**泛化能力强**：仅用 12K 训练数据，可处理 128K 上下文（10× 训练窗口，100× 平均样本长度）。
- **可插拔增强**：无缝集成到 A-MEM、LightMem、MemoryOS 中，带来 +13.5%~+20.5% 的绝对提升。
- **时序检索能力**：对“firstly”“lastly”等时间关键词敏感，相似度曲线与段索引呈单调/尖峰关系，t-SNE 可视化显示空间分离。

## 7. 优点
- **方法论创新**：从表示层面解决动态检索问题，而非依赖复杂流水线（重写/重排序/多轮推理），简洁且高效。
- **训练高效**：仅用 184k 样本（< 1% 其他模型数据量），标准 SFT 即可超越多阶段训练的 competitors。
- **无需额外 token 开销**：记忆在编码阶段构建，不依赖生成器，不增加推理 token 消耗。
- **轻量级适配**：LoRA 低秩适配即可学到演化表示，参数效率高（即使 rank=16 性能稳定）。
- **开源友好**：提供 0.8B/2B/4B 系列，兼容白盒和黑盒 API。项目页面和代码已公开。

## 8. 不足与局限
- **域外退化**：受限于计算资源和训练数据规模，在完全 out-of-domain 场景可能性能下降。
- **缺少多模态**：当前不支持视觉/音频等模态；虽然架构可扩展，但跨模态记忆需要更大队列和维度对齐，留作未来工作。
- **编码速度**：相比可并行编码的静态模型，顺序处理慢（22s vs 3~10s），虽然峰值显存更低（20.9GB vs 32.3~69.3GB）。实时性要求高的场景可能需要进一步优化。
- **实验覆盖**：主要聚焦对话/个性化/长文档检索，未在通用 MTEB 等静态检索基准上报道结果（论文强调长上下文场景）。但作为该领域专门工作，此局限可接受。
- **验证语言**：仅在英文上实验，多语言能力未经测试。
