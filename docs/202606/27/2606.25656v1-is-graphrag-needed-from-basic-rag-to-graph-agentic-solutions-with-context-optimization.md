## 摘要
随着GraphRAG和Agentic RAG等高级RAG变体的出现，一个关键问题是如何以及何时使用它们。本文针对半结构化知识库，引入了不同RAG场景的评估与比较框架，包括常规RAG、GraphRAG、模块化RAG和Agentic RAG。我们提供了9种标准化RAG场景的实现，并通过实验进行全面比较。这些场景针对真实用例设计，考虑了数据和领域限制，涵盖从简单的基于文档的检索到高级功能，如文本-图混合检索、与计算或预定义领域知识图的集成、智能体多步规划以及智能体-图融合。此外，我们提出了一种用于GraphRAG和Agentic RAG的新颖上下文工程方法，解决了上下文/记忆溢出问题，通过新的表示形式和智能体循环设计高效管理文本和图检索，使token使用量减少19%-53%。进一步分析揭示了一个检索-生成差距：扩展的检索并不能按比例提高生成质量，表明面向检索的指标夸大了高级检索的收益。本研究为构建生产级智能RAG系统提供了何时以及如何使用这些方法的数据驱动见解。

## Abstract
As advanced RAG variants like GraphRAG and Agentic RAG emerge, one leading question is when and how to use them. Here, we introduce a framework for different RAG scenarios evaluation and comparison on semi-structured knowledge bases, including regular RAG, GraphRAG, Modular RAG and Agentic RAG. We provide implementation for 9 standardized RAG scenarios, and conduct experiments for a comprehensive comparison. These scenarios are designed for real use cases regarding data and domain restrictions, spanning from simple document-based retrieval to advanced features such as hybrid text-graph retrieval, integration with computed or pre-defined domain knowledge graphs, agentic multi-step planning, and agent-graph integration. Besides, we present a novel context engineering method for GraphRAG and Agentic RAG, addressing the context/memory overflow issues, efficiently managing text and graph retrievals with new representations and agentic loop design, leading to 19%-53% reduction on token usage. Moreover, further analysis identifies a retrieval-generation gap where expanded retrieval does not proportionally improve generation quality, suggesting retrieval-oriented metrics overstate advanced retrieval benefits. This work provides data-driven insights on when and how to use them for building production-ready intelligent RAG systems.

## 3. 实验设计

- **数据集**：STaRK-Prime（精确医学领域半结构化知识库），包含约129K个实体（疾病、药物、蛋白质、基因等）的文本描述文档，以及来自PrimeKG的810万条知识图关系。使用官方测试集（人类生成的查询）。
- **基准/评估指标**：采用生成导向的评估指标，即LLM生成自然语言答案并返回实体ID集合后，与ground truth比较：Hit@1、Hit@5、Recall@20（R@20）、Mean Reciprocal Rank（MRR）。此外，增加Recall of Retrieval（RoR）衡量检索覆盖率，以及平均Token使用量（#Tokens）用于上下文优化实验。
- **对比方法**：9种RAG场景相互比较；在GraphRAG场景下对比不同超参数（最大路径数、子图数）；在Agentic场景下对比是否启用思考、是否增加知识图工具、是否应用上下文优化等变体。
- **基础设施**：
  - 预定义知识图：使用Amazon Neptune Analytics，通过LlamaIndex加载，CypherTemplateRetriever进行子图提取（2跳邻居，最大100/200/500路径）。

  - Agentic RAG框架：Strands Agents（默认ReAct循环）。
  - LLM：Anthropic Claude 3.7 Sonnet（所有LLM组件统一使用）。

## 6. 主要结论与发现

- **是否真的需要GraphRAG？**：答案高度依赖上下文，需要细致分析。
  - 对于复杂关系查询，GraphRAG有潜力；但简单的1跳关系文档增强（场景2）竟优于更复杂的GraphRAG（场景5），表明简单方法结合关系信息即可获得高性价比。
  - 自主Agentic RAG（场景8，仅检索工具）在所有方法中表现最优（Hit@1 0.6881），说明LLM作为Agent能够自主实现复杂推理，无需过多专门工具。但增加知识图工具（场景9）反而性能下降。
- **检索-生成差距**：关键发现——扩展检索并未按比例提升生成质量。即使检索覆盖率从77.5%提升至83.5%，LLM答案召回率仅徘徊在45%-48%。原因包括：
  - 位置注意力衰减（“Lost in the Middle”）：越靠后的实体提取率越低。
  - LLM偏好规范/显著实体而非穷举枚举。
  - 查询语法（如单数提问“What is...”）诱导单答案预期，压制多实体提取。
- **上下文优化有效**：实现19%-53%的token减少，同时维持或提升性能。批量Agentic检索虽减少工具调用次数并提高检索覆盖率，但最终生成上下文膨胀导致生成质量下降，表明迭代式Agent推理在检索与生成之间起到了关键的信息过滤作用。

## 7. 优点

- **系统性框架**：首次在同一框架下对9种RAG场景进行公平比较，覆盖从基础到高级的完整光谱，为实践者提供直接选型指导。
- **上下文工程创新**：提出关系分组图表示、去重策略、批量Agentic检索等实用技术，解决生产部署中的token成本瓶颈。
- **揭示检索-生成差距**：通过位置分析和案例研究，深入解释为何检索指标（Hit@k、MRR）夸大了实际收益，推动RAG评估向生成导向转变。
- **实验丰富且透明**：大量消融、超参数探索和深度分析表格，支持结论的可信度。

## 8. 不足与局限

- **数据集单一**：仅使用STaRK-Prime（精确医学领域），结论在其他领域（如法律、金融）的泛化性未知。

- **评估指标有限**：缺乏事实忠实性、幻觉率、回答流畅性等质量维度评估；仅依赖检索驱动的指标（尽管是生成后的）。
