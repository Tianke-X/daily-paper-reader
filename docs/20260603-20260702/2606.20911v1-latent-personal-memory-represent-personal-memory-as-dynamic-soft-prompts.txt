Personalizing large language models (LLMs) requires encoding long-term, user-specific be-havioral patterns in a way that is computation-ally efficient, scalable, and compatible with a frozen base model. We present Latent Personal Memory (LPM), a scalable framework that rep-resents user-specific history as a compact, per-sistent matrix of N latent slots, that are inter-pretable. A shared cross-attention projection network maps these slots into dynamic, input-conditioned soft prompts that are prepended to the input of a frozen LLM. We evaluate LPM on PersonaMem v1 and LoCOMO bench-marks across Qwen3-1.7B, 4B, and 8B back-bones. Results demonstrate that LPM outper-forms LoRA and Prompt Tuning by up to 8.8 % and 54.4% in overall accuracy respectively on PersonaMem v1, while reducing KV-cache usage by over 64x . On LoCoMo, LPM matches LoRA’s accuracy with 120x fewer trainable pa-rameters. We also show that LPM’s efficiency grows with context length and outperforms full-context at 128K context length. 

1 Introduction & Background 

Personalizing large language models (LLMs) re-quires retaining long-term user-specific informa-tion while remaining computationally efficient ,scaling with a growing user base, minimizing stor-age, and preserving the model’s general capabili-ties. Existing approaches struggle to jointly satisfy these requirements. Full-context methods (Liu et al., 2025) prepend the entire user history at in-ference, incurring high latency and large KV-cache overhead due to the quadratic cost of self-attention. 

RAG approaches (Lewis et al., 2020; Guo et al., 2025; Huang and Huang, 2026; Xia et al., 2025)

Recent text-based mem-ory methods (Hu et al., 2026) store history as dis-crete structured units that are retrieved and updated as preferences evolve, but often rely on agentic architectures with large memory footprints with multiple LLMs, hierarchical memories, and ded-icated modules for memory updates (Wang and Chen, 2025; Wang et al., 2026; Yan et al., 2026). Parameter-efficient fine-tuning methods like LoRA (Hu et al., 2022) train small per-user weights but modify the base LLM, risking degra-dation of general reasoning (Shuttleworth et al., 2025). Prompt tuning (Lester et al., 2021) learns static per-user soft prompts but suffers from limited expressivity . Context-compression methods (Hu-ber et al., 2025; Tan et al., 2024) target generic compression rather than long-term personalization. 

Latent memory methods instead encode user in-formation into the model’s internal representations. Recent latent memory architectures with richer in-ternal memory (Wang et al., 2024, 2025; Yang et al., 2024; Zhang et al., 2026) require large-scale or continual pretraining to learn memory read/write dynamics making them neither scalable nor inter-pretable . 2 Related Works 

Full-context methods. A direct approach to per-sonalization is to prepend the user’s entire inter-action history to each query (Liu et al., 2025). While conceptually simple, this incurs high la-tency and substantial computational overhead from large K-V cache sizes. The cost stems from the self-attention mechanism, whose complexity scales quadratically with sequence length, so inference costs grow rapidly with the number of input tokens. 

Retrieval-augmented generation (RAG). RAG approaches (Lewis et al., 2020; Guo et al., 2025; Huang and Huang, 2026; Xia et al., 2025) reduce prompt length by retrieving only relevant snippets from an external store. However, they rely on per-user indices that grow with history length and incur additional overhead for updates as new interactions accumulate. 

Text-based memory methods. A growing line of work (Hu et al., 2026) stores user history as discrete units in structured form, retrieving, reorganizing, and updating entries as preferences evolve. Many of these adopt agentic architectures with high mem-ory footprints, requiring multiple LLMs in memory, hierarchical memory levels, and dedicated modules for memory evolution and generation (Wang and Chen, 2025; Wang et al., 2026; Yan et al., 2026). A smaller subset, including Mem0 (Chhikara et al., 2025) and LightMem (Fang et al., 2026), instead targets scalable, efficient deployment with lower latency. All of these methods, however, leave the user memory external to the LLM, maintaining a per-user index that grows with history length and preserves user history in the text space. 

Parameter-efficient fine-tuning. Methods such as LoRA (Hu et al., 2022) train a small set of per-user weights efficiently. However, because they directly modify the base LLM’s weights, they risk degrading its general reasoning capabilities as ob-served in some works (Shuttleworth et al., 2025). Latent memory methods. An alternative line of work encodes user information directly into the model’s internal representations, offering a more compact alternative to text-based memory. Prompt tuning (Lester et al., 2021) learns static per-user soft prompts concatenated before the question, compressing each user into a fixed, low-capacity representation that limits personalization expressiv-ity. Context-compression methods (Huber et al., 2025; Tan et al., 2024) train compressors that map long contexts into shorter soft-token sequences, but target generic compression rather than long-term personalization. More recent latent memory ar-chitectures introduce richer internal memory struc-tures at higher cost: MemoryLLM (Wang et al., 2024) and M+ (Wang et al., 2025) learn a self-updatable latent memory pool but require large-scale C4 / Common-Crawl-style pretraining to ac-quire their read/write dynamics. Memory3 (Yang et al., 2024) represents latent memory as K-V cache vectors written and read via a continual pretraining stage. MemGen (Zhang et al., 2026) generates la-tent memory tokens via a separate generator LLM, adding both parameter and GPU-memory costs. LPM also falls in the latent memory family but focuses specifically on efficient memory represen-tation for personalization . We build on a frozen pre-trained LLM, requiring no additional pretrain-ing , and pair a small per-user slot matrix with a shared projection network trained once across the user population. This yields a memory footprint that scales efficiently with the number of users. 

3 LPM Methodology 

LPM builds on the observation that parameter-efficient fine-tuning (PEFT) methods offer better accuracy and lower computational costs than few-shot In-Context Learning (ICL) (Liu et al., 2022). Rather than storing or re-processing raw conver-sation histories at inference time, Latent Personal Memory (LPM) represents each user’s history as a compact matrix of learned latent slots. A shared projection network then maps these slots into dy-namic, input-conditioned soft prompts that are prepended to the input of the frozen LLM.

3.2 Inference. 

At inference, LPM uses only the user’s question to retrieve soft prompts, which are prepended to the question tokens and passed to the frozen LLM, yielding a short effective context and low latency (Fig 1 (b)). The cost of processing each user’s his-tory is paid once during training and amortized over all subsequent inference queries, making LPM sig-nificantly more efficient than full-context or RAG-based approaches at deployment. 

4 Experiments 

4.1 Datasets 

We evaluate LPM on two long-context personaliza-tion benchmarks: PersonaMem v1 (Jiang et al., 2025), a multiple-choice personalization bench-mark, and LoCOMO (Maharana et al., 2024), a generative long-context QA benchmark. 

PersonaMem v1. PersonaMem v1 (Jiang et al., 2025) is a long-range personalization benchmark containing approximately 32K-token contexts per user together with multiple-choice questions span-ning six in-situ personalization categories: fac-tual recall, suggesting new ideas, tracking pref-erence evolution, reasons behind preference up-dates, preference-aligned recommendations, and generalization to new scenarios. We evaluate on all 20 users in the dataset and report multiple-choice accuracy. The per-category question counts are summarized in Table 9. 

LoCOMO. LoCOMO (Maharana et al., 2024) is a long-conversation benchmark consisting of multiple dialogue sessions between two individ-uals with average context lengths of approximately 9K tokens. The benchmark contains single-hop, multi-hop, temporal, and open-domain questions requiring long-context reasoning over conversa-tions. Following prior work, we use GPT-5-mini as a judge model (Chhikara et al., 2025) to eval-uate free-form generations against gold answers using binary CORRECT/WRONG scoring. The per-category question counts are summarized in Table 8. Alongside task performance, we report infer-ence efficiency metrics : context length , KV-cache memory , peak GPU memory , latency , and trainable parameters .

Experimental Setup. We evaluate on Qwen3-1.7B, 4B, and 8B backbones. Considering realistic deployment, we assume access only to long user histories Cuser without user-specific QA supervi-sion. Unlike context distillation approaches (Snell et al., 2022) that rely on synthetically generated supervision, we avoid generating QA pairs from the context, as such methods introduce addi-tional computational overhead and are difficult to scale . 6 Analysis of Learned Memory Representations 

A key advantage of our approach is that the train-able memory slots learn user-specific representa-tions. To verify this, we perform a geometric anal-ysis on the PersonaMemv1. For each of the 20 users, we average the trained slot matrices across 8 heads to obtain Mu ∈ R32x256 , then project all slot vectors into 2D with UMAP (McInnes et al., 2020) (Figure 3). The slots form coherent, well-separated per-user clusters, suggesting they cap-ture user-specific behavioral structure. To test whether this geometric separation reflects mean-ingful behavioral differences, we conduct a rubric-based qualitative similarity study on representative users (2, 4, 14, 15). For each pair, the full con-versational histories ( ∼32K tokens) were given to CLAUDE -O PUS -4.7 as judge, which scored pair-wise similarity along dimensions including domain-specific preferences, communication style, person-ality traits (rubric in Table 11; full protocol in Ap-pendix B). Table 7 reports representative scores. Notably, all pairings involving User 14 (the far-thest from other users in UMAP space) receive the lowest similarity scores, aligning the geometric and rubric-based views. Example judge outputs for the most and least similar pairs appear in Ta-bles 13and 14. Further qualitative findings are in Appendix 2.3. Together, these results indicate that the learned memory slots are interpretable and en-code behavioral traits beyond topical content. 7 Conclusion 

In this paper. we introduce LPM, a scalable frame-work for LLM personalization that represents each user’s history as a small persistent matrix of la-tent slots and projects them into soft prompts via a shared projection network, making it scalable. Across PersonaMem and LoCOMO, LPM consis-tently outperforms LoRA and Prompt Tuning with 

fewer parameters and lower inference costs . Geo-metric analysis of the learned memory slots reveals that LPM learns interpretable latent representa-tions that capture personalized behavioral patterns. 

Limitations 

Our work is not without its limitations. 

(i) Hybrid approaches: LPM under-performs in memorizing factual knowledge. Future works can explore hybrid approaches combining latent and text-based memory approaches for personaliza-tion. A complementary explicit fact cache may be needed for applications that require exact recall. 

(ii) Memory granularities: LPM relies on the fact that all heads equally capture long-term and short-term memory. Structuring memory heads to capture long-term vs. short-term memory of the user using heuristics, can be a future direction of research.

1.2 Chat Template and Training Details 

We train LPM with a chunk-based objective that operates over a chat-formatted sequence. The soft prompts produced by the memory module are in-serted into the first system block, immediately after the system token and before the " ### CONTEXT END ### " marker that delimits the latent context from the textual instruction. The training loss is com-puted only from the first context token. All tokens preceding it (including the soft-prompt positions, the system instruction, and the user instruction) are masked out and do not contribute to the loss. This ensures that the model is optimized to use the la-tent memory rather than to reconstruct the template scaffolding around it. The exact chat template used during training is reproduced below. Two aspects of this template are worth highlight-ing. Firstly, the soft prompts are placed inside the 

system block rather than as a separate prefix, which keeps them within the region of the sequence that the underlying chat model is trained to treat as con-ditioning. Secondly, the user instruction is kept short and generic to anchor the assistant block but not encode any task-specific information.

1.3 LOCOMO Evaluation Judge Prompt 

For evaluation on LOCOMO we follow the proto-col introduced Mem0 (Chhikara et al., 2025) and use an LLM-as-a-judge setup. Each generated answer is compared against the gold answer by an GPT-5-mini that returns a binary CORRECT / WRONG label together with a short rationale. The judge is prompted to be lenient with respect to sur-face form (e.g. verbosity, paraphrase, alternative date formats) and to treat refusals as correct when the gold answer itself indicates that the information is not available in the conversation history. We pro-vide both the system prompt and the per-example user prompt below for reproducibility. 

System prompt 

You are an LLM Judge. Your task is to label an answer to a question as CORRECT or WRONG . You will be given the following data: 0. a question (posed by one user to another user), 0. a “gold” (ground truth) answer, 0. a generated answer, which you will score as CORRECT /WRONG .The point of the question is to ask about something one user should know about the other user based on their prior conversations. The gold answer will usually be a concise and short answer that includes the referenced topic, for example: 

Question: Do you remember what I got the last time I went to Hawaii? 

Gold answer: A shell necklace The generated answer might be much longer, but you should be generous with your grading—as long as it touches on the same topic as the gold answer, it should be counted as CORRECT .For time-related questions, the gold answer will be a specific date, month, year, etc. The generated answer might be much longer or use relative time references (like “last Tuesday” or “next month”), but you should be generous with your grading—as long as it refers to the same date or time period as the gold answer, it should be counted as CORRECT . Even if the format differs (e.g., “May 7th” vs. “7 May”), consider it CORRECT if it is the same date. There is an edge case where the actual answer cannot be found in the data, and in that case the gold answer will say so (e.g., “You did not mention this information.”); if the generated answer says that it cannot be answered or it does not know all the details, it should be counted as CORRECT .Respond with JSON: {"reasoning": "...", "correct": true or false}
