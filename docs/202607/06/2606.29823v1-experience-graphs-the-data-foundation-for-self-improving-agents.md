## 摘要
数据库社区通过认识到新工作负载需要新系统架构，不断推动技术进步。我们认为，长周期智能体任务——代码生成、科学发现、硬件设计——正是这样一种工作负载。这些智能体进行探索：它们生成工件、执行工具、观察失败、分支和修复，跨越数百步。这种搜索产生了一个结构化的对象，我们称之为经验图谱：可执行工件、工具输出、奖励、兄弟比较和因果谱系。然而，现有的智能体框架将这种经验视为一次性状态——JSON检查点和会话日志，在崩溃后无法恢复，无法跨用户查询，也无法物化为训练数据。我们提出Trellis：一个数据基础，将经验图谱视为一等公民、受治理、可查询的数据库状态。核心洞察是：对经验图谱的搜索是一种数据库访问模式。前沿选择是一个查询，跨会话重用是向量引导的图检索，训练数据提取是一个物化视图，重建智能体在过去任何步骤的知识是一个时间旅行查询。当数据库拥有经验图谱时，智能体变为无状态计算，崩溃恢复、水平扩展和闭环训练飞轮成为架构的副产品。我们将该设计落地于Meta的生产加速器内核优化器KernelEvolve，其中跨会话重用实现了约10倍的目标加速，同时token成本降低52%。更广泛地说，Trellis将推理时搜索从一次性计算转变为持久的机构资产：日志使数据库可靠；经验图谱可能使智能体累积。

## Abstract
The database community has repeatedly advanced the state of the art by recognizing that new workloads demand new system architectures. We argue that long-horizon agentic tasks -- code generation, scientific discovery, hardware design -- are such a workload. These agents explore: they generate artifacts, execute tools, observe failures, branch, and repair over hundreds of steps. This search produces a structured object we call an experience graph: executable artifacts, tool outputs, rewards, sibling comparisons, and causal lineage. Yet existing agent frameworks treat this experience as disposable state -- JSON checkpoints and session logs that cannot be recovered after a crash, queried across users, or materialized into training data. We propose Trellis: a data foundation that treats the experience graph as first-class, governed, queryable database state. The core insight is that search over experience graphs is a database access pattern. Frontier selection is a query, cross-session reuse is vector-seeded graph retrieval, training-data extraction is a materialized view, and reconstructing what an agent knew at any past step is a time-travel query. When the database owns the experience graph, agents become stateless compute, and crash recovery, horizontal scaling, and a closed-loop training flywheel emerge as architectural byproducts. We ground the design in KernelEvolve, a production accelerator-kernel optimizer at Meta, where cross-session reuse reaches a target speedup roughly 10x faster at 52% lower token cost. More broadly, Trellis turns inference-time search from disposable computation into a durable institutional asset: logs made databases reliable; experience graphs may make agents cumulative.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：长周期智能体任务（如代码生成、科学发现、硬件设计、安全研究）不再是单次推理，而是逐步迭代搜索的过程。智能体会生成工件、执行工具、观察失败、分支修复，持续改进，产生一个结构化的“经验图谱”——包括可执行工件、工具输出、目标奖励、兄弟比较、因果谱系和可变搜索统计。
- **问题**：现有智能体框架（如Claude Code、Codex、Meta的KernelEvolve早期实现）将经验仅作为一次性状态存储（JSON检查点、会话日志），崩溃后无法恢复、无法跨用户查询、无法直接用于训练、无法实现跨会话复用。这种状态无法支撑递归自我改进的闭环训练飞轮。
- **核心含义**：经验图谱不应是“知识”的存储（如声明性/程序性/情景记忆），而应是“搜索尝试记录”的持久化、可查询、受治理的数据库状态。将经验从一次性计算转变为持久的机构资产，是智能体获得累积性学习能力的关键基础设施。

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：将智能体系统设计为**双循环架构**：
    - **内循环**：技能驱动的智能体会话（生成→执行→评估→持久化），生成单个节点。
    - **外循环**：搜索编排（如MCTS、进化搜索、贪心策略），决定下一个扩展节点，基于经验图谱选择前沿、组装上下文、插入节点、更新奖励。
    - **持久数据基础**：将所有探索状态存储在持久化、可查询的数据库中，使智能体进程变为无状态的服务计算。
- **Trellis数据基础**：
    - **逻辑模型**：四表层级模式——Tasks（问题定义）、Sessions（搜索会话）、Nodes（每次尝试：父链接、工件、执行输出、适应度、搜索统计）、Prompt Histories（LLM输入输出消息）。大对象存储在对象存储，表只保留引用。
    - **上下文作为受管状态**：子节点可以继承父节点会话，或通过祖先查询重建上下文。`prompt_history`表支持时间旅行重建节点当时的完整上下文。
    - **物理架构**：存储与计算分离。使用Axiom优化器将SQL、Cypher图查询、向量检索融合为单一物理计划，路由到不同后端（操作存储、向量索引、列式数仓）。
    - **向量引导的图检索**：先通过向量相似度找到相似历史任务，再沿着关系链接（任务→会话→节点）图遍历，获取最高适应度的子图。这实现了跨会话复用。
    - **并发多智能体搜索**：多个工作节点可并行探索同一会话，但需处理MCTS反向传播时的原子性更新。论文指出最终一致性足够，但提出了数据库新研究问题。
    - **多版本状态与时间旅行**：字段级变更日志按逻辑步号记录，支持重建任意历史状态（训练时需重构智能体当时实际看到的状态，避免未来信息泄露）。
    - **训练数据作为物化视图**：SFT轨迹（根到叶高适应度路径）、DPO偏好对（兄弟节点比较）、GRPO组（从持久缓冲区采样，多次单步扩展替代完整重演）。训练数据不是事后抓取，而是数据库查询的结果。
- **闭环飞轮**：生产探索填充存储 → 存储改善检索与前沿选择 → 轨迹通过SFT/DPO/GRPO训练模型 → 改进模型产生更高质量尝试 → 飞轮累积。

## 6. 主要结论与发现

- **跨会话记忆显著加速收敛**：p=0.1和p=0.5均在约5步内达到1.2倍加速，而无记忆基线需要51步——**10倍加速**。
- **降低buggy节点率**：无记忆基线有55%的buggy节点（触发昂贵调试循环）；p=0.1降至34%，p=0.5降至21%。这直接导致每有效节点的token成本降低**52%**。
- **探索-锚定权衡**：高注入率（p=0.5）收敛快但多样性降低（仅8种策略组合，基线有20种），且最终最佳解不如无记忆基线（1.36× vs. 1.49×）。平衡设置p=0.1保留大部分收敛增益并产生最多高质量解（≥1.3×）。
- **数据库内建恢复与扩展**：崩溃恢复成为自然的数据库查询操作，无需额外检查点机制；水平扩展只需增加/移除工作节点。
- **训练数据即查询**：无需重新遍历整个搜索树，SFT/DPO/GRPO数据可从存储中直接提取，减少多轮展开的时间成本。
