## 摘要
顺序演化的LLM记忆使得智能体能够复用过往经验，但现有系统通常直接部署每个局部生成的记忆更新，而不检查其是否改进了未来行为。结果，有助于当前任务的更新可能覆盖有用知识、引入过度具体的规则，或使最终记忆偏向近期示例。我们提出Janus，一个插件式记忆控制器，决定是否接受候选记忆更新或保留先前记忆。为使决策高效，Janus使用记忆动量触发器来识别记忆更新轨迹中的可疑偏差，并在一个紧凑的混合评估集（覆盖性、边界性和新鲜性任务）上比较新旧记忆，而非重放完整历史。Janus是方法无关的，可包裹现有更新器而不改变其更新规则。在六个数据集、两个骨干LLM和两个记忆更新器上，Janus相对于对应的基础更新器将平均准确率提升了+2.7至+4.6个百分点。

## Abstract
Sequentially evolving LLM memory enables agents to reuse past experience, but existing systems usually deploy each locally generated memory update without checking whether it improves future behavior. As a result, updates that help the current task may overwrite useful knowledge, introduce over-specific rules, or bias the final memory toward recent examples. We propose Janus, a plug-in memory controller that decides whether to accept a candidate memory update or retain the previous memory. To make this decision efficient, Janus uses a Memory Momentum Trigger to identify suspicious deviations in the memory-update trajectory, and compares old and new memories on a compact hybrid evaluation set of coverage, boundary, and fresh tasks instead of replaying the full history. Janus is method-agnostic and wraps existing updaters without changing their update rules. Across six datasets, two backbone LLMs, and two memory updaters, Janus improves average accuracy by +2.7 to +4.6 points over the corresponding base updaters.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：现有顺序演化LLM记忆系统（如ExpeL、DC-RS）在每次任务后都会根据当前反馈生成记忆更新，并**不加筛选地直接部署**。这种做法存在根本性风险：
  - 局部有益的更新可能**覆盖此前积累的有用知识**；
  - 可能引入**过度具体或有噪声的任务特定规则**；
  - 最终记忆会**偏向后出现的任务/示例**，导致未来任务性能非单调下降（如图1所示）。
- **背景**：LLM智能体在顺序任务中通过外部记忆复用经验，但已有系统只关注“如何生成记忆”，忽视了“是否该部署该更新”这一关键决策。因此需要一种**记忆验证机制**，在部署前评估候选更新的全局效用。
- **核心问题**：如何高效判断何时需要比较新旧记忆、以及用什么任务进行比较，从而在保持低回放成本的同时做出正确的部署决策。

## 2. 方法论：核心思想与关键技术细节

### 2.1 整体思想
- **Janus**是一个**插件式控制器**，包裹任意基础记忆更新器（如ExpeL、DC-RS），在每次候选更新产生后，决定**接受候选记忆**还是**保留前序记忆**，不改变基础更新器的内部规则。
- 将记忆更新视为**部署决策**：只有在新记忆可能提升最终记忆对未来任务的有用性时才替换。

### 2.2 Memory Momentum Trigger (MMT)
- **目的**：决定**何时**触发新旧记忆比较，避免每次任务都比较（高成本）或固定间隔（可能错过有害更新）。
- **技术细节**：
  - 使用文本编码器 ϕ(·) 将记忆状态映射为向量。
  - 计算候选更新方向：\( z_t = \phi(\hat{M}_t) - \phi(M_{t-1}) \)
  - 维护更新方向的指数移动平均：\( m_t = \beta m_{t-1} + (1 - \beta) z_t \)
  - 当当前更新方向与历史动量方向的**余弦相似度**低于阈值 τ 时，触发比较：\( \cos(z_t, m_{t-1}) < \tau \)
  - **直觉**：连续相似的更新代表渐进式精炼，无需比较；方向突变可能意味着引入有用知识或有害改写，需要验证。

### 2.3 混合触发时评估集 (Hybrid Trigger-Time Evaluation Set)
- **目的**：当触发比较后，用**紧凑的任务子集**近似全历史回放，决定新旧记忆哪个更好。
- **构成**：\( E_t = S_t^{\text{cov}} \cup S_t^{\text{bdry}} \cup F_t \)
  - **Coverage Set (覆盖集)**：通过聚类所有已见任务，选取离每个聚类中心最近的任务，代表全局分布。
  - **Boundary Set (边界集)**：记录那些**记忆选择导致正确性变化**的任务（即新旧记忆在该任务上给出不同正确结果）。这些任务对记忆选择高度敏感。
  - **Fresh Set (新鲜集)**：从**上次触发后**遇到的新任务中采样，防止评估集过时。
- **更新规则**：每次触发后，覆盖集重新聚类；边界集收集新翻转任务并去重；新鲜集从近期任务中采样。
- **决策**：在 E_t 上分别用新旧记忆运行LLM，选择准确率更高的记忆部署。

### 2.4 算法流程（文字描述）
1. 对于第 t 个任务，基础更新器生成候选记忆 \(\hat{M}_t\)。
2. Janus计算更新方向 z_t 和动量 m_t。
3. 若 \(\cos(z_t, m_{t-1}) \ge \tau\)（未触发），直接接受候选记忆。
4. 若触发，构建混合评估集 E_t（含覆盖、边界、新鲜任务）。
5. 在 E_t 上评估 M_{t-1} 和 \(\hat{M}_t\)，选择准确率更高的记忆作为 M_t。
6. 更新覆盖集、边界集、新鲜集及动量。

## 6. 主要结论与发现

- **Janus持续提升基础记忆更新器的最终记忆有用性**：在DC-RS上平均提升+2.7至+4.6个百分点，在ExpeL上类似提升。
- **MMT触发的时机具有重要意义**：与随机或固定间隔触发相比，Janus在相同触发预算下获得更高准确率，甚至有时超过“每次触发”策略（如DC-RS on GPQA），说明过度比较反而可能引入噪声。
- **混合评估集的所有三个组成部分（覆盖、边界、新鲜）均对最终性能有贡献**，其中新鲜集最为关键，表明依赖静态支持集会过时。
- **Janus使得记忆性能在任务流上更稳定上升**，而基础更新器在后期常出现性能下降（图4）。
- **超参数敏感性较低**：默认K=20和τ=0.0在平衡性能与成本方面表现良好；增加支持集大小未必提升性能。

## 7. 优点

- **方法无关**：Janus可包裹任意现有的提示式记忆更新器，无需修改其内部规则，即插即用。
- **高效**：MMT大幅减少了不必要的比较（触发率受数据集和阈值控制，如HumanEval仅12%），混合评估集避免了全历史回放，控制了回放成本。
- **设计简洁**：动量方向和余弦相似度易于计算，阈值τ和动量系数β提供清晰控制。
- **稳健**：在多个数据集、两个LLM、不同顺序下均一致提升，且对超参数不敏感。
- **可解释性**：定性案例清晰展示了Janus能够拒绝“正确但狭窄”的更新，接受“引入可重用能力”的更新，行为合理。
